///==================GAME MANAGER SCRIPTS================
GameManager.cs - Centralised master controller for the game.

Purpose:
Acts as the main singleton that holds global variables, player state flags, references to key scene objects, and shared utility functions. Almost every other gameplay script will reference this to check or update game-wide states. Ensures that important information (like whether the player is currently wearing glasses, in the middle of Tai Chi, or allowed to interact with certain objects) is consistent across all scripts and scenes.

Key Responsibilities:
- Holds references to critical scene objects: player rig, left and right hands, cameras, fade panels, models for male/female characters, glasses, dentures, etc.
- Tracks game-wide interaction flags such as:
  toPickUpPhone (waiting for player to pick up phone)
  toPutGlassesOn / toTakeDenturesOff (waiting for these specific actions)
  toDoTaiChi (ready to start Tai Chi sequence)
  toStartSpasming (trigger illusion behaviour in furniture)
  toLookAtObjective (ready for a look-based event to be completed)
- Manages fade-in/fade-out transitions for scene changes using fadePanel’s Animator.
- Handles display of on-screen alerts through ShowAlert(...) and HideAlert() functions.
- Provides utility methods used by other scripts, including:
  DebugLog(string) – sends debug text to in-game console via DebugTextTest.
  IsPlayerWithinPosition(...) – checks if the player is inside a specified 3D area, often used for triggering scenario steps when the player walks into a zone.
  FindPiecesToFlip(...) and AnimateFlippingPieces(...) – special logic for the Othello/checkers style mini-games.

How it interacts with other scripts:
- Scenario managers (e.g., ScenarioManagerPresentBad/Good/ReneeTest) call GameManager methods to show alerts, trigger fades, and set player interaction flags.
- AlertTextController relies on GameManager to initiate alert messages.
- ControllerInteractionsManager checks GameManager’s flags before allowing/disallowing certain interactions.
- PostProcessingController can be toggled via GameManager to simulate blurred vision when glasses are off.
- TaiChiManager sequences can be triggered through references held in GameManager.

Usage:
- Only one GameManager should exist in the scene (singleton pattern).
- All scene objects that need centralised control should be assigned in the Inspector.
- When creating new gameplay steps, set/clear the appropriate GameManager flags so other systems respond correctly (e.g., set toPickUpPhone = true before showing the phone’s prompt).

ScenarioManagerPresentBad.cs - sceneToPlay enum is used to select where the flow of event starts from (Bathroom/Bedroom)
-> narration_1 & narrationAudioClips_1 is for Bathroom+LivingRoom Scene / narration_2 & narrationAudioClips_2 is for Bedroom Scene
-> drop glasses sound effect can play together with the narration clip (In function DropGlassesReaction(), line 196)
Purpose:
Controls narrative flow for the Present-Bad scenario (Bathroom and Bedroom). Manages glasses, dentures, cane, teleportation, and illusion mechanics.
Scenes:
Bathroom: Starts with washing, cane interaction, glasses, and a phone call.
Bedroom: Handles hallucinations, medication interaction, and ending sequence.
Key Functions:
PlaySegment1Part1() to Part3_3(): Bathroom narrative sequence (wash, grab cane, pick up phone, glasses interaction, phone call).
DropGlassesReaction(): Plays sound and triggers alert when glasses are dropped.
PlaySegment2Part1() to Segment2Part2_3(): Bedroom hallucination sequence.
FirstMedicationDropped() / SecondMedicationDropped(): Triggers scripted animations and narration.
StaredAtPhoto(): Final memory trigger, leads to fade out and scene transition.
StopPrevDialogue(): Stops current coroutine and hides alert.
PlayAudioAndNarration(...): Utility functions for showing alerts + playing audio with variations.

ScenarioManagerPresentGood.cs - Purpose:
Controls flow for the Present-Good scenario (Bathroom → Living Room → Void Deck). Coordinates narration, events, teleportation, and NPC movement.
Scenes:
Bathroom/LivingRoom: Interact with bathroom items, answer phones, and interact with friends.
Voiddeck: Involves Tai Chi, checkers, mic interaction, karaoke.
Key Functions:
PlaySegment1Part1() to Part5(): Guides player from bathroom to answering phone to leaving house.
PlaySegment2Part1() → MovingFromTaichiToCheckers(): Tai Chi to checkers transition.
PlayCheckersTransition() → PlaySecondPieceCaptured(): Manages checkers interactions.
MovingFromChessToKaraokeCorner(): Transition after checkers win.
PlayMicOnFace() / PlayMicOffFace(): Mic interaction logic for karaoke.
PlayEndOfScenario(): Ends scene with fade and goodbye screen.
StopPrevDialogue(), PlayAudioAndNarration(...): Utility methods for alert/audio flow.

ScenarioManagerReneeTest.cs - Test scenario manager for “PastNegative” type scenes (Hawker Bathroom → Hawker Stall).
SceneToPlay Enum: Determines if flow starts from Bathroom or Stall.
HawkerPartOne():
Displays washing up prompt.

Sets PlayerTeleport.MoveToToiletDoor = true.
HawkerPartTwo():
Triggered after bathroom segment, intended to move player towards hawker stall.

Usage:

Can be expanded to add narration/audio sequences like ScenarioManagerPresentBad and PresentGood.

ScenarioPromptManager.cs - Handles loading and displaying on-screen prompts for specific scenario/scene combinations from a CSV file.
CSV Structure: First row is headers; each subsequent row contains ScenarioID, SceneID, and the prompt text.
Prompt Display: Uses AlertTextController to show prompts with optional duration and “stay on screen” mode.
Key Functions:
ShowPrompt(...): Displays a specific prompt by scenario, scene, and index.
GetAllPrompts(...): Returns all prompts for a given scenario and scene (for sequencing logic).
ChangeScene(...): Updates internal SceneID tracking.
Usage in Project:
Attach to a manager object in your scene.
Assign the CSV file and AlertTextController in the Inspector.
Call ShowPrompt(...) from scenario managers when you want a prompt to appear in-game.

///==================REDUNDANT SCRIPTS================
VelocityEstimator.cs - Legacy component from Valve’s VR code used for estimating slice velocity.
Was originally tied to the EzySlice system for determining slice direction/force.
Currently redundant unless you are re-enabling real-time slice detection instead of manual slicing triggers.

SliceLimit.cs - Small helper script for EzySlice.
Tracks remaining number of cuts a sliced object can take.
Used in older slicing logic; no longer needed if objects aren’t meant to be sliced multiple times.

SlicedObject.cs - Implements slicing via the EzySlice library.
Uses VelocityEstimator and SliceLimit to control slice direction and max cuts.
Adds Rigidbody + MeshCollider to sliced pieces and applies explosion force for effect.
Status in Project: Redundant unless mesh slicing is reintroduced — the EzySlice system is more for physical mesh cutting, not currently central to gameplay.

ChoppingBlockController.cs - Custom interaction for locking the player’s left hand to an item on a chopping block.
Intended for a chopping/cutting minigame.
Works by detecting when ControllerInteractionsManager.leftHandAnchor enters a trigger and snapping to a leftHandSnapPoint.
Status: Currently unused; could be adapted for fixed-position hand interactions in future scenes.

DebugTextTest.cs - In-game debug console overlay.
Shows Unity Debug.Log output inside VR so you can test without the editor console.
Useful for validating logic in standalone builds.
Intended to be improved to display logs on interaction with a test object.
///==================REDUNDANT SCRIPTS================

SmoothPivotRotator.cs - Utility for smoothly rotating an object to a target rotation, with optional auto-return.
Common Use: Animated props (fans, levers) or subtle scene motion.
Can be triggered via script or UnityEvent with StartDefaultRotation() / StartRotationTo(...).
autoReturn can reset rotation after a delay.

PlayerTransparency.cs - Adjusts transparency of listed models based on player’s camera pitch angle.
Example: Fade the player body model when looking down to prevent obstruction.
Usage: Assign any player model GameObjects with SkinnedMeshRenderer to the models list.
